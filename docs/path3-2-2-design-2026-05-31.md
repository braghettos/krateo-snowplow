# Path 3.2.2 — Walker Pagination Design (0.30.220)

## Pivot from (B)+(C) to walker extension

Original brief proposed:
- (B) informer-driven `populateWidgetContentL1`
- (C) cohort-level RBAC-sensitive widget seed

Diego's 2026-05-31 directive (`feedback_prewarm_follows_frontend`):
> "walking INIT is how prewarm works"

(B)+(C) HALTED. The fix is to extend the F2 INIT walker so it traverses
per-composition widget CRs that today it bypasses.

## Empirical confirmation of the gap

From `/debug/vars` on 0.30.219 helm rev 378:

- `snowplow_phase1_walk_children`: walker visited only `2 buttons + 10 markdowns
  + 42 panels` per-composition widget CRs out of ~85K total in cluster
- Per-composition coverage comes from `compositions-page-datagrid` resolving with
  `perPage=5, page=1` defaults → 5 rows × 2 datagrids = 10 compositions walked
- Customer-driven `widget-content-miss-per-user-fallback`: buttons=3,437,
  markdowns=3,401 → 99.98% of per-composition widget CRs are cold at first hit

## Root cause

`phase1_walker.walk()` at `phase1_walk.go:872` resolves each widget exactly
ONCE at `(perPage, page)`. For apiRef+`resourcesRefsTemplate` widgets the
resolver fans the template over the apiRef RA's resolved data, sliced by
`(page, perPage)`. The compositions-page-datagrid produces only 5 of ~50K
rows because the walker passes `perPage=5, page=1` and never calls again
with `page=2`. The downstream per-composition widget CRs (panels, buttons,
markdowns) are therefore never reached, never resolved, never populated
in `widgetContent` L1.

The resolver itself signals more pages exist:
`status.resourcesRefs.slice.continue = true` when `tot >= perPage`
(resolve.go:124-134). The walker reads `status.resourcesRefs.items[]` but
ignores `.slice.continue`.

## Fix — `walkApiRefPages` page iteration

At the existing walker's post-resolve site (phase1_walk.go:989), AFTER
`populateWidgetContentL1` Puts the page-1 envelope, IF the widget is
apiRef+`resourcesRefsTemplate`-driven AND `.slice.continue == true`, the
walker continues with `page=2, 3, ...` until:
- `.slice.continue == false` OR the resolved item list is empty, OR
- `page > PREWARM_APIREF_MAX_PAGES` (default 500 → at perPage=5, covers
  up to 2,500 rows per apiRef widget; configurable via env)

For each additional page resolved:
- The walker Puts the envelope under the matching content L1 key
  (key tuple from declared slice or PREWARM_PAGE_LIMIT, symmetric with
  current page-1 logic)
- The walker recurses into each NEW child ref the page produces

## Data-driven; no hardcoded GVRs

The page iteration fires on the SHAPE predicate `isApiRefTemplateDriven`:
both `spec.apiRef.name != ""` AND `spec.resourcesRefsTemplate` non-empty.
This is the SAME shape `shouldSkipEmptyWidgetShell` and
`isRBACSensitiveApiRefWidget` already key on (per
`feedback_no_special_cases`). No widget-name, no GVR literal.

The walker discovers per-composition GVRs (buttons / markdowns / panels)
by parsing each child ref's `/call?` Path via the EXISTING
`util.ParseCallPathToObjectRef` — exactly the existing recursion mechanism.

## Bounding the cost

- **Total widgets paginated**: only the apiRef+resourcesRefsTemplate
  widgets reachable from INIT roots. At today's cluster:
  `compositions-page-datagrid` (~50K compositions) + `blueprints-page-datagrid`
  (small). One pagination per such widget at boot.
- **Per-widget pages**: bounded by `PREWARM_APIREF_MAX_PAGES`. At
  500 pages × perPage=5 = 2,500 rows materialised per apiRef widget.
  Real cluster has ~50K compositions ⇒ ~10,000 pages needed for full
  coverage. **PHASE 0 ships with default 500 (2,500 compositions)** — we
  prove the mechanism + measure refresher amplification before raising
  the cap.
- **Per-composition recursion**: unchanged from today — `walkShouldRecurse`
  gate (verb=="GET"), `visited` cycle-set, `phase1MaxWalkDepth` cap. So
  each new composition reached spawns the same depth-bounded subtree it
  always did.
- **Refresher amplification** (the 0.30.185 HARD REVERT lesson): the
  resulting L1 entries are visible to the refresher just like any other
  entry. Phase 0 cap=500 keeps the populated-entry count to
  ~2,500 compositions × ~5 widgets ≈ 12,500 new L1 entries → manageable
  delta vs today's baseline. We instrument before raising.

## Feature flag posture (Diego mandate: NO new env vars / flags)

Per the dispatch brief's hard rules section: "No new env vars / flags".
The pagination rides on the EXISTING `cache.PrewarmEnabled()` gate (the
same flag that controls whether `Phase1Warmup` runs at all). When prewarm
is on, pagination runs. When prewarm is off, the entire walker is bypassed
by the caller (main.go). This is consistent with
`project_single_cache_flag_direction` — one flag `CACHE_ENABLED` end-state.

The per-widget max-pages bound is a Go constant
(`phase1MaxApiRefPages`) — configurable by code change, not env. If the
ship needs raising, it lands as a code edit + tag; not a runtime knob.

## Files modified

1. `internal/handlers/dispatchers/phase1_walk_pagination.go` — NEW.
   `iterateApiRefPages(ctx, w, in, gvr, depth, page, perPage, keyPerPage, keyPage)`:
   after page 1 resolved, if predicate fires + `.slice.continue == true`,
   loop page=2..MaxPages, re-fetching a fresh CR copy per page, calling
   `populateWidgetContentL1` per page, recursing into each new child ref.
3. `internal/handlers/dispatchers/phase1_walk.go` — single call site after
   the existing `populateWidgetContentL1(...)`/`extractResourcesRefsItems`
   pair: `iterateApiRefPages(...)`.

## Tests

- `phase1_walk_pagination_test.go`:
  1. Predicate-on widget with `.slice.continue==true` triggers page 2..N.
  2. Predicate-off widget with `.slice.continue==true` does NOT paginate.
  3. Max-pages cap honoured: walker stops at `PREWARM_APIREF_MAX_PAGES`.
  4. Feature-flag off: zero pagination calls.
  5. Each additional page Puts its envelope into widgetContent L1.
  6. Cycle-set (visited) honoured across pages — same child ref on
     page 2 that appeared on page 1 is not re-walked.
