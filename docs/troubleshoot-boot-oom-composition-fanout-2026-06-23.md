# Troubleshooting — snowplow 1.4.3 boot-OOM (concurrent composition fan-out) — 2026-06-23

**Cluster:** bs-test-ger-03 (operations-dev-krateo-io, europe-west10), snowplow 1.4.3, read-only via `/tmp/bs-test-ger-03.kubeconfig`. Diagnosis TRACED to canonical code @ 1.4.3 + live runtime artifacts (`/tmp/bs-sp-prev.log`, expvar `http://34.32.39.90:8081/debug/vars`).

## Symptom
snowplow OOMKilled at boot (exitCode 137, 15:32:22Z, ~8min into boot-1; limit 8Gi). Recovered on restart, stable since. **Transient heap spike, not a leak** — expvar: HeapAlloc=60MiB, HeapReleased=7431MiB, HeapSys=7508MiB, StackSys=3.2MiB, NumGC=383 (Go grabbed ~7.5GiB at peak, GC freed it all; arena high-water persists; off-heap/stack ruled out).

## Root cause: unbounded, recursion-multiplied COLD fan-out with NO concurrent-resolve memory bound
A synchronous concurrent burst at boot (745 stage-dispatches in the single second 15:31:52; dominant stage `allCompositionResources`) — the 5 `fsa-y*` composition-resources pages loaded cold simultaneously. Three stacked amplifiers on the customer `/call` path:
- **A1 — per-resolve `dict` + double-encode.** The serve `dict` (resolve.go:243) accumulates every stage output, held until `Resolve` returns (resolve.go:1508); then `encodeResolvedJSON` (restactions.go:241) materializes a full second byte copy. Each in-flight composition resolve holds dict + encoded bytes.
- **A2 — cold child fetches, no informer, apistage cache OFF.** The `allCompositionResources` step iterates `.getComposition.status.managed` (fsa-y8 = 26 resources) + GETs each. Those GVRs (services/deployments/clusters/…) have NO synced informer → `dispatchViaInformer` not-servable (informer_dispatch.go:438/516) → live apiserver fetch. expvar: `…services|informer-fallthrough-not-servable:1810, deployments:1777, clusters:845`. AND `RESOLVED_CACHE_APISTAGE_ENABLED` was UNSET → `ApistageL1Enabled()==false` → the per-K8s-call content cache (apistageContentServe, apistage.go:423) is bypassed → every load re-fetches all children cold. (As of the #57 fold the api-stage L1 is on-by-default under `RESOLVED_CACHE_ENABLED` — `ApistageL1Enabled()==ResolvedCacheEnabled()`, resolved.go:510 — so this gate is now open whenever the resolved cache is on.)
- **A3 — `resolve:true` recursion multiplier.** `.status.managed` has 4 nested RESTActions + ~13 widgets/composition; the step's `resolve:true` recurses each via `maybeResolveInProcess`→`ResolveNestedCall` (nested_call.go:85/162/198) into a full nested resolve + its own fan-out, up to `nestedCallMaxDepth=8` (nested_call_depth.go:31). Burst log shows "nested resolve depth limit exceeded (8)". So one composition `/call` = 26 × (nested resolve trees).
- **THE DEFECT — no bound.** No global cap on concurrent customer `/call` resolves, no cross-resolve memory budget. `iterParallelism` (resolve.go:57, g.SetLimit ≤32, resolve.go:1301) bounds per-stage width but is re-applied PER nested resolve (multiplicative). `markCustomerInFlight` (restactions.go:77) is a bare counter (makes prewarm yield, not a semaphore). ΣN concurrent composition resolves × (A1×A2×A3) > 8Gi.
- **GOMEMLIMIT=7GiB couldn't hold** (it IS set): a soft limit makes GC aggressive but cannot evict a genuinely-large simultaneously-REACHABLE set; the burst's live set itself exceeded 7GiB.

## Fix design (layered; ship (b)+(a) together, (c) fast-follow)
- **(b) root-cause:** the apistage content cache absorbs the 26 child GETs; 2nd+ concurrent loaders + reloads hit warm cells, the `not-servable` cold-refetch collapses. **Delivered by the #57 fold** — api-stage on-by-default under `CACHE_ENABLED`+`RESOLVED_CACHE_ENABLED` (`ApistageL1Enabled()==ResolvedCacheEnabled()`); no per-feature `RESOLVED_CACHE_APISTAGE_ENABLED` to set. Still MUST land with (a) so the auto-on cannot itself cold-fill-spike — see F-FOLD-5: the fold's only runtime activations are (i) the apistage content cache and (ii) the cluster-list prewarm IF a harvester is present (independently gated on `PrewarmEnabled() && PrewarmContentEnabled()`, phase1_walk.go:493), so it does NOT turn on an un-semaphored boot populate burst. (Open Q: registering informers for the managed GVRs is the deeper fix — scope/scale decision, flagged not picked.)
- **(a) hard safety net [~40-60 LOC + -race test]:** process-wide weighted concurrency semaphore on the customer resolve entry (restactions.go ServeHTTP, acquire before Resolve at :220), sized EMPIRICALLY (per-entry peak dict+encode bytes × safety; budget = fraction of GOMEMLIMIT — feedback_capacity_caps_empirical_per_entry_cost). Preserves customer-priority (admit customer ahead of prewarm; serialize excess customer load → slower not OOM — feedback_customer_priority_over_refresher). Prior art: prewarm hotSem / x/sync/semaphore weighted.
- **(c) fast-follow [~20-30 LOC]:** cap the `resolve:true` recursion multiplier on the customer path (env/context-scoped `nestedCallMaxDepth`, or count nested breadth against the (a) budget). No per-resource special-case.
